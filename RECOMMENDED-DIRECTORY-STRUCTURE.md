# 推奨ディレクトリ構造ガイド

Claude Code を日常的に使う前提で、ローカルのフォルダ構成をどう設計するかをまとめたガイドです。「なんとなく `Documents/` に全部置いている」状態から卒業し、AI エージェントの能力を最大限引き出せる構造を作ることを目指します。

## 1. なぜディレクトリ構造が重要か

Claude Code は `CLAUDE.md` という指示ファイルを **ディレクトリ階層に沿って自動で読み込む** 仕組みを持っています。つまり「どのフォルダで `claude` を起動したか」によって、AI に渡るルール・メモリ・コンテキストが変わります。

逆に言えば、ディレクトリ構造を雑に作ると：

- 業務中に個人プロジェクトのルールが混ざる
- 無関係な指示が AI に渡って挙動が不安定になる
- メモリやスキルが領域横断で汚染される

**ディレクトリの単位 = Claude に与えるコンテキストの単位** と捉えて設計するのがコツです。

---

## 2. 推奨する3層構造

ホームディレクトリ直下を、以下の3つの領域に分けます。

```
~/
├── Workspace/       # 仕事・業務コード（会社リポジトリ、業務ツール）
│   ├── repos/
│   ├── scripts/
│   └── CLAUDE.md    # 業務文脈・社内ルール
│
├── Personal/        # 個人プロジェクト・個人管理物
│   ├── task-manager/
│   ├── experiments/
│   └── CLAUDE.md    # 個人ルール
│
└── .claude/         # Claude Code のグローバル設定
    ├── CLAUDE.md    # 全プロジェクト共通ルール（人格・セキュリティ等）
    ├── commands/    # スキル（スラッシュコマンド）
    └── projects/    # セッション履歴・メモリ
```

### それぞれの役割

| 領域 | 用途 | CLAUDE.md に書く内容 |
|------|------|---------------------|
| `~/.claude/` | 全プロジェクト共通の人格・ポリシー | セキュリティ、口調、普遍的な作業ルール |
| `~/Workspace/` | 業務関連のすべて | 社内ツール、業務文脈、組織固有の慣習 |
| `~/Personal/` | 個人プロジェクト・私物 | 個人の好み、サイドプロジェクトの方針 |

`CLAUDE.md` は階層に沿って **上書きされていく** ので、`~/.claude/CLAUDE.md` の普遍ルールの上に、各領域の固有ルール、さらに各リポジトリ固有のルールが積み上がる構造になります。

## 3. 設計原則

### 原則1: 業務と個人を物理的に分ける

最大の理由は **コンテキスト汚染の防止**。業務中に個人プロジェクトのメモリや指示が混ざると邪魔ですし、その逆も同じです。Claude のメモリ機能は階層に沿って参照されるため、領域を分けるだけで自然に分離できます。

### 原則2: リポジトリは深くネストしない

```
~/Workspace/repos/<project>/      ✅ 推奨（2層）
~/Workspace/work/2025/q1/<proj>/  ❌ 非推奨（深すぎ）
```

`Glob` / `Grep` の検索が速く、`CLAUDE.md` の自動ロードも素直に効きます。

### 原則3: experiments / sandbox を1個用意する

```
~/Personal/experiments/
```

「ちょっと試したい」「Claude に投げてみたい」用の捨て場を用意しておくと、本筋のプロジェクトが散らかりません。終わったらフォルダごと捨てる前提で運用します。

### 原則4: 設定ファイルの居場所を一元化する

| 種類 | 推奨パス |
|------|---------|
| グローバル設定 | `~/.claude/settings.json` |
| グローバル指示 | `~/.claude/CLAUDE.md` |
| グローバルスキル | `~/.claude/commands/` |
| プロジェクト指示 | `<project>/CLAUDE.md` |
| プロジェクトスキル | `<project>/.claude/commands/` |

## 4. 他人への説明の仕方

「なぜホーム直下を `Workspace` と `Personal` に分けているの？」と聞かれたとき、以下のように説明すると Claude Code を知らない相手にも伝わりやすいです。

> ホーム直下を用途別のルートに分けています。`Workspace` は業務、`Personal` は個人の管理物やサイドプロジェクト。Claude Code は `CLAUDE.md` という指示ファイルをディレクトリ階層に沿って読み込むので、業務文脈と個人文脈を物理的に分けておくと、AI に渡る情報・ルール・メモリが混ざらず、それぞれに最適化した振る舞いをさせられます。全プロジェクト共通の設定は `~/.claude/` に置いて、領域ごとのルールはそれぞれのルートに置く、という3階層構造です。

ポイントは「**AI 前提の合理性**」として説明すること。単なる整理整頓ではなく、エージェントの挙動を制御するための設計だと伝えると納得されやすくなります。

## まとめ

- ディレクトリの単位 = Claude に与えるコンテキストの単位
- `~/.claude/` / `~/Workspace/` / `~/Personal/` の3層構造を基本にする
- リポジトリは深くネストしない、`experiments/` を1つ用意する
- 他人に説明するときは「AI のコンテキスト分離のため」と伝える
