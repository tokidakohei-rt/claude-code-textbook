# Chapter 4: パイプラインSkills - 複数のSkillを繋げる

## 単発Skill → パイプラインへ

Vol.1 では `/report` という単発の Skill を作りました。1つのSkillで1つの処理をする形です。

Vol.2 では、**複数の Skill を順番に繋げて、大きなタスクを段階的に処理する** 考え方を学びます。これを「パイプライン」と呼びます。

## このリポジトリのパイプライン

```
/research [テーマ]    →  Web検索で情報収集
        ↓
/analyze [調査結果]   →  収集した情報を分析
        ↓
/report [分析結果]    →  レポートを生成
        ↓
/review [レポート]    →  フィードバック
```

各 Skill は **入力と出力が明確** に定義されています:

| Skill | 入力 | 出力 |
|-------|-----|------|
| `/research` | テーマ（キーワード） | `output/research-xxx.md` |
| `/analyze` | 調査結果ファイル | `output/analysis-xxx.md` |
| `/report` | 調査結果 + 分析結果 | `output/report-xxx.md` |
| `/review` | 任意のファイル | 画面にフィードバック表示 |

> **ポイント**: 前の Skill の出力が、次の Skill の入力になる。この「繋がり」がパイプラインの本質です。

## パイプライン設計の3原則

### 1. 各Skillは1つの責務に絞る

```
悪い例: /do-everything  → 検索して分析してレポート作って全部やる
良い例: /research → /analyze → /report  → 段階ごとに分ける
```

なぜ分けるか:
- 途中で止めて修正できる
- 一部だけやり直せる（分析だけ再実行など）
- 各段階の品質を個別にチェックできる

### 2. 入出力を明確にする

各 Skill ファイルに書くべきこと:
- **何を入力として受け取るか**（`$ARGUMENTS` の意味）
- **何を出力するか**（ファイルパス、フォーマット）
- **出力のフォーマット**（Markdownの構造）

### 3. 中間ファイルを残す

パイプラインの各段階でファイルを保存します:

```
output/
├── research-生成AI.md    ← 調査結果
├── analysis-生成AI.md    ← 分析結果
└── report-生成AI.md      ← 最終レポート
```

中間ファイルがあれば:
- どの段階で問題が起きたか特定しやすい
- 分析だけやり直したい場合、調査からやり直す必要がない
- 別のテーマと結果を比較できる

## Skill ファイルの読み解き

実際に `/research` Skill の中身を見てみましょう:

```
.claude/commands/research.md を読んで
```

注目ポイント:
1. `$ARGUMENTS` でテーマを受け取る
2. 調査項目が5つ明確に定義されている
3. 調査のルールが書かれている（複数ソース、出典付き等）
4. 出力フォーマットが固定されている
5. 保存先パスが指定されている

この構造があるから、誰が実行しても **同じ品質・同じ形式** の調査結果が出ます。

## 自分でSkillをカスタマイズする

既存の Skill を改良してみましょう。例えば `/research` に以下を追加できます:

- 「日本語と英語の両方で検索する」というルールを追加
- 調査項目に「規制・法的環境」を追加
- 出力に「信頼度（高/中/低）」の列を追加

> **ポイント**: Skill は Markdown ファイルなので、テキストを書き換えるだけでカスタマイズできます。プログラミングは一切不要です。
