# Chapter 7: レポート生成とレビュー

## レポートを生成する

いよいよ最終成果物です。ここまでの調査・分析結果を統合して、業界研究レポートを生成します。

### `/report` の実行

```
/report output/research-[テーマ].md output/analysis-[テーマ].md
```

レポートが `output/report-[テーマ].md` に生成されます。

### レポートの構成（テンプレート）

生成されるレポートは `templates/report-template.md` の構成に従います:

1. **タイトル・調査概要** — テーマ、調査日、サマリー
2. **市場概要** — 規模、成長率、全体像
3. **主要プレイヤー** — 企業名、ポジション、特徴
4. **トレンド分析** — 最新の動向と背景
5. **機会とリスク** — 成長ドライバーとリスク要因
6. **示唆** — So What? この業界について最も重要なこと
7. **出典** — 参考にした情報源のリスト

## レビューと改善のサイクル

### 1回目のレビュー

```
/review output/report-[テーマ].md
```

フィードバックをもとに改善します:

```
上のレビューを踏まえて、output/report-[テーマ].md を改善してください。
特に [指摘された点] を重点的に修正してください。
```

### 2回目のレビュー

改善したら、もう一度レビューを受けましょう:

```
/review output/report-[テーマ].md
```

> **ポイント**: 「作る → レビュー → 改善 → レビュー」のサイクルを回すことで、品質が段階的に上がります。これは実際の仕事でも同じです。

## `/review` の種明かし

ここまで使ってきた `/review` Skill の中身を見てみましょう:

```
.claude/commands/review.md を読んで
```

実は、ただの **Markdown ファイル** です。

中に書かれているのは:
- 評価観点（構成、情報の質、具体性、読みやすさ、過不足）
- フィードバックのルール（良い点を先に、改善は3つまで、点数はつけない）
- 出力フォーマット

つまり、**あなたは今日、自分だけのレビュアーを「設計する」力をすでに持っている** ということです。

## レビュアーをカスタマイズする

`/review` を自分の仕事に合わせてカスタマイズしてみましょう。

### 例1: プレゼン資料用のレビュアー

評価観点を変える:
- ストーリーの流れ
- 1スライド1メッセージになっているか
- 図表の活用度

### 例2: メール文面のレビュアー

評価観点を変える:
- 用件が冒頭で明確か
- 相手への配慮
- 誤字脱字・敬語の適切さ

### 例3: 議事録のレビュアー

評価観点を変える:
- 決定事項が明記されているか
- 次のアクションと担当者が書かれているか
- 抜け漏れがないか

> **ポイント**: Skills の Markdown を書き換えるだけで、どんな業務にもフィットするレビュアーが作れます。これが Claude Code を「使いこなす」ということです。

## 今日学んだことのまとめ

| 機能 | 何ができるようになったか |
|------|----------------------|
| WebSearch / WebFetch | Web上の最新情報を取得して活用する |
| Hooks | ツール実行に連動した自動処理を設定する |

## Vol.1 + Vol.2 で身についた力

```
Vol.1: 手元の情報を → 整理して → 構造化されたアウトプットにする
Vol.2: Webの情報を → 調べて分析して → レポートにまとめてレビューする
```

この2つを組み合わせれば、ほとんどの知的作業を Claude Code で効率化できます。
